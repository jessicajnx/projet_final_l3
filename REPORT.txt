================================================================================
RAPPORT DE PROJET DEVOPS - AUTOMATISATION DE DEPLOIEMENT
================================================================================


1. CONTEXTE ET OBJECTIFS
================================================================================

Ce projet a pour objectif la conteneurisation et l'automatisation du déploiement 
d'une application web full-stack. L'enjeu principal est de réduire les erreurs 
manuelles lors de la mise en production, d'accélérer les cycles de déploiement 
et de garantir la cohérence entre les environnements de développement et de 
production.

L'application déployée est composée de trois services interdépendants : une 
interface utilisateur frontend servie par Nginx, une API backend développée 
avec Node.js et Express, et une base de données PostgreSQL pour la persistance 
des données.


2. ARCHITECTURE ET CHOIX TECHNIQUES
================================================================================

2.1 STRUCTURE APPLICATIVE

Frontend : Nginx 1.27-alpine servant une application HTML/JavaScript statique
Rôle : Interface utilisateur, reverse proxy vers les endpoints API
Communication : Proxy HTTP vers le backend sur le port 3000

Backend : Node.js 20 avec framework Express
Rôle : API REST avec endpoints de santé et métier
Endpoints : GET /health (vérification du statut), GET /api/message (retour de message)
Dépendances : express, pg (PostgreSQL client), dotenv

Base de données : PostgreSQL 16-alpine
Rôle : Persistence des données applicatives
Initialisation : Script init.sql créant les tables nécessaires


2.2 TECHNOLOGIES DE CONTENEURISATION

Docker : Chaque service dispose d'un Dockerfile optimisé
Backend : Build multi-étapes (stage dépendances + stage runtime)
Frontend : Image Nginx avec configuration reverse proxy et fichiers statiques
Avantages : Isolation des services, reproductibilité, déploiement rapide


2.3 ORCHESTRATION LOCALE

Docker Compose 1.31+ : Orchestration multi-services en environnement local
Fichier principal : docker-compose.yml avec 3 services (db, backend, frontend)
Configuration : Volumes pour la persistence PostgreSQL, healthchecks, variables 
d'environnement, mapping de ports
Avantages : Développement facile, test complet de la stack avant déploiement cloud


2.4 ORCHESTRATION CLOUD

Kubernetes (K3s 1.34.6) : Distribution légère de Kubernetes pour la VM Azure
Manifests : Namespace, ConfigMap, Secret, Deployments (frontend/backend), 
Services, PersistentVolumeClaim pour la base de données
Avantages : Scalabilité, auto-healing, orchestration production-grade
Note : K3s choisi pour les ressources limitées de la VM (847 MB RAM initial)


2.5 INTÉGRATION CONTINUE ET DÉPLOIEMENT CONTINU

GitHub Actions : Pipeline de CI/CD automatisé sur chaque push vers main
Étapes : Tests unitaires → Build images Docker → Push Docker Hub → 
Déploiement SSH sur VM
Secrets managés : Credentials Docker Hub, clé SSH privée, mot de passe PostgreSQL


2.6 REGISTRE IMAGE

Docker Hub : Stockage centralisé des images conteneurisées
Namespace : linuxmint75/backend et linuxmint75/frontend
Avantages : Partage facile, intégration CI/CD native, versioning automatique


3. DÉPLOIEMENT ET ÉTAPES RÉALISÉES
================================================================================

3.1 PHASE 1 : PRÉPARATION ET DÉVELOPPEMENT LOCAL

Première étape : Création de la structure complète du projet en local avec les 
trois services (frontend, backend, db).

Tests backend : Implementation de tests unitaires avec Jest et Supertest 
validant les endpoints /health et /api/message. Les 3 tests passent avec 
succès, couvrant le fonctionnement normal et les cas d'erreur (base de données 
indisponible).

Dockerfiles : Création d'images optimisées pour chaque service, avec attention 
particulière au multi-stage build pour le backend afin de réduire la taille 
de l'image finale.

Docker Compose local : Configuration d'un fichier docker-compose.yml permettant 
de lancer les 3 services avec les bonnes variables d'environnement, volumes, 
et healthchecks. Vérification que l'application complète fonctionne localement 
avec la communication frontend → backend → base de données établie.


3.2 PHASE 2 : CONFIGURATION CI/CD

Pipeline GitHub Actions : Mise en place d'un workflow automatisé 
(.github/workflows/ci-cd.yml) déclenché à chaque push sur la branche main.

Job "tests" : Exécution des tests Jest du backend. Le pipeline s'arrête si 
les tests échouent, garantissant la qualité du code livré.

Job "build-and-push" : Construction automatique des images Docker pour 
backend et frontend, puis push vers le registre Docker Hub (linuxmint75 namespace).

Job "deploy" : Connection SSH à la VM Azure, récupération du code à jour, 
application des manifests Kubernetes avec kubectl, et rollout des pods pour 
intégrer les nouvelles images.


3.3 PHASE 3 : DÉPLOIEMENT SUR VM CLOUD AZURE

Infrastructure : Mise à disposition d'une VM Ubuntu 24.04 sur Azure 
(adresse IP 132.164.81.42) avec 847 MB RAM initial et possibilité de configuration.

Installation des outils : Docker, Docker Compose plugin, K3s (distribution 
allégée de Kubernetes) installés sur la VM via script setup-vm.sh.

Configuration K3s : Installation en mode allégé (flags --disable traefik 
--disable servicelb --disable metrics-server) pour minimiser la consommation 
de ressources.

Application des manifests : Déploiement de la stack applicative via kubectl 
apply -k k8s/ (utilisation de Kustomize pour la composition).

Exposition de l'application : Configuration d'endpoints Kubernetes (Services 
NodePort) pour accéder aux services via l'IP VM.


3.4 PHASE 4 : ACCÈS PUBLIC ET STABILISATION

Accès via port 30080 : Initialement, l'application était exposée via le 
NodePort 30080 (http://132.164.81.42:30080).

Problèmes de stabilité K3s : La VM avec 847 MB RAM s'est avérée insuffisante 
pour K3s stable. Erreurs récurrentes incluant "TLS handshake timeout", 
"Unable to connect to the server", et "kubelet stopped posting node status".

Tentatives de correction : Ajout d'un swap de 2 GB, réinstallation de K3s 
en mode allégé, mais instabilité persistante.

Pivot vers Docker Compose : Pour assurer la stabilité de l'application en 
production, passage à Docker Compose directement sur la VM 
(docker-compose.vm.yml avec exposition sur port 80).

Accès public sécurisé : L'application est maintenant accessible via 
http://132.164.81.42 (port 80), autorisation par défaut sur Azure pour 
les ports HTTP/HTTPS.

Vérification : L'endpoint /health répond avec le statut JSON 
{"status":"ok","service":"backend","db":"up"}, confirmant le fonctionnement 
complet de la stack.


4. GESTION DES VARIABLES D'ENVIRONNEMENT ET SECRETS
================================================================================

Variables d'environnement :
- API_MESSAGE : Message retourné par l'endpoint /api/message
- DB_HOST, DB_USER, DB_PASSWORD, DB_NAME, DB_PORT : Paramètres PostgreSQL
- NODE_ENV : Environnement (development/production)
- PORT : Port d'écoute du serveur backend

Fichier .env.example : Template fourni pour la configuration locale
Fichier .env : Non commité dans Git (ajouté à .gitignore)

Secrets GitHub Actions :
- DOCKERHUB_USERNAME : Identifiant Docker Hub (linuxmint75)
- DOCKERHUB_TOKEN : Token personnel pour l'authentification Docker Hub
- VM_HOST : Adresse IP ou domaine de la VM
- VM_USER : Utilisateur SSH (azureuser)
- VM_SSH_KEY : Clé privée SSH (format PEM) pour accès sans mot de passe

Secret Kubernetes :
- POSTGRES_PASSWORD : Mot de passe administrateur PostgreSQL cité dans la 
VM après déploiement, stocké en tant que Secret Kubernetes (non en dur dans 
les manifests)

ConfigMap Kubernetes :
- Configuration applicative globale (API_MESSAGE, noms de bases de données, etc.)

Bonnes pratiques respectées :
- Secrets jamais committés en dur
- Séparation entre configuration (ConfigMap) et données sensibles (Secret)
- Utilisation des GitHub Secrets pour les identifiants CI/CD
- Templates fournis (.env.example, secret.example.yaml) pour faciliter la 
configuration

6. DIFFICULTÉS RENCONTRÉES ET RÉSOLUTIONS
================================================================================

6.1 INSTABILITÉ K3S SUR VM AZURE

Problème initial : La VM fournie (847 MB RAM) s'est avérée sous-dimensionnée 
pour exécuter K3s de manière stable. Les erreurs incluaient des timeouts TLS 
lors de la communication avec l'API Kubernetes, des déconnexions HTTP/2 client, 
et l'indisponibilité du kubelet.

Logs observés : "Unable to connect to the server: net/http: TLS handshake 
timeout", "Timeout: Too large resource version", "client connection lost"

Cause root : Mémoire insuffisante → kubelet arrête de poster le statut du 
nœud → liveness probes échouent → pods non déployés → stack instable

Tentative 1 : Ajout d'un swap de 2 GB via dd et mkswap. Résultat : légère 
amélioration temporaire, mais instabilité persistante.

Tentative 2 : Réinstallation de K3s en mode allégé avec flags 
--disable traefik --disable servicelb --disable metrics-server pour réduire 
l'empreinte mémoire. Résultat : Problèmes similaires, K3s incompatible avec 
< 1.5 GB RAM opérationnel.

Tentative 3 : Augmentation RAM demandée à l'équipe (faible succès initial)

Résolution pragmatique : Pivot vers Docker Compose native sur la VM sans 
Kubernetes. Avantages : Plus léger en mémoire, exécution stable et immédiate, 
toujours production-ready pour une petite application. Les manifests 
Kubernetes restent disponibles pour déploiement sur une VM plus grande.

Leçon apprise : Kubernetes nécessite minimum 2-4 GB RAM pour une instance 
production. Pour les ressources < 2 GB, Docker Compose est une solution 
pragmatique et robuste.


6.2 ACCÈS RÉSEAU À LA VM

Problème : Connexion timeout lors des tentatives d'accès au port 8081 
(frontend) et 3000 (API) depuis l'extérieur.

Cause : Azure Network Security Group (NSG) bloque par défaut tous les ports 
sauf 22 (SSH), 80 (HTTP), 443 (HTTPS).

Résolution : Exposition via port 80 (HTTP standard) en utilisant 
docker-compose.vm.yml override file, autorisant l'accès public direct.

Résultat : L'application est maintenant accessible sur 
http://132.164.81.42/health et http://132.164.81.42/


6.3 TESTS ET VALIDATION

Problème surfacé lors des tests locaux : Le backend devait valider la 
connexion à la base de données dans l'endpoint /health avant de retourner 
le statut.

Résolution : Implémentation d'une requête de test PostgreSQL (SELECT 1 FROM 
healthcheck;) et gestion appropriée des erreurs avec status codes 503 
(service unavailable) si DB down.

Résultat : L'endpoint /health valide correctement l'état de la stack complète.

9. CONCLUSION
================================================================================

Le projet DevOps a atteint ses objectifs principaux :

1. Conteneurisation complète : Tous les services (frontend, backend, base de 
données) sont conteneurisés et optimisés.

2. Orchestration fonctionnelle : La stack fonctionne localement via Docker 
Compose et sur le cloud via Docker Compose (pragmatique) ou prépositionnée 
pour Kubernetes sur infra plus grande.

3. Automatisation CI/CD : Pipeline GitHub Actions garantit que chaque push 
exécute tests, builds images et déploie automatiquement.

4. Accès public : L'application est accessible en production sur 
http://132.164.81.42 avec les trois services communicants et la base de 
données opérationnelle.

5. Documentation complète : README, templates de configuration, scripts 
d'installation fournissent tous les éléments pour reproduire l'installation.

Défis gérés : L'instabilité de K3s sur ressources limitées a été résolue par 
un pivot pragmatique vers Docker Compose, maintenant la stabilité tout en 
documentant les bonnes pratiques Kubernetes pour déploiement sur infra 
dimensionnée.

Recommandations pour l'avenir :

- Augmenter les ressources VM (2-4 GB RAM minimum) pour une utilisation 
production de Kubernetes.
- Mettre en place une surveillance (Prometheus, Grafana) pour tracked la 
santé de la stack.
- Implémenter des tests d'intégration pour valider les communications 
inter-services.
- Ajouter des logs centralisées (ELK stack, Loki) pour le debugging.
- Configurer des alertes sur les endpoints de santé pour la détection 
proactive de pannes.

Cette solution DevOps démontre les compétences clés nécessaires pour automatiser 
et déployer des applications web modernes dans un environnement cloud, en 
balançant les choix théoriques (Kubernetes) avec les réalités pratiques 
(ressources limitées).
